# purl2repo v2.0.0

`purl2repo` 2.0.0 is a production-focused rewrite of the package URL to
repository resolver. It replaces prototype-era return dictionaries with typed
models, a stable JSON contract, explicit confidence, evidence, warnings, and a
host-aware repository model.

## Highlights

- Python 3.14 package using a modern `pyproject.toml` and `src/` layout.
- Public Python API with `parse_purl`, `resolve`, `resolve_repository`,
  `resolve_release`, and reusable `Resolver`.
- Typer CLI with `parse`, `resolve`, `repo`, `release`, `supports`, and
  `version` commands.
- Full metadata-backed support for PyPI, npm, Cargo, Maven, NuGet, and Go.
- Direct repository handling for `pkg:github` and `pkg:bitbucket`.
- Artifact hub handling for Hugging Face and MLflow.
- Generic PURL support through explicit `vcs_url`, `repository_url`, and
  `download_url` qualifiers.
- Repository URL validation, conservative release-link resolution, and bounded
  fallback scraping only when structured metadata is insufficient.
- GitHub, GitLab, Bitbucket, and generic git host normalization.
- Unit and live integration test separation with coverage enforcement.
- CI includes Ruff, format check, mypy, tests, build verification, CLI smoke
  test, and pinned Trivy dependency scanning for high and critical
  vulnerabilities.

## Breaking Changes

- Loose dictionary outputs are replaced with typed dataclasses and stable JSON
  serialization.
- CLI usage is now command-based: `parse`, `resolve`, `repo`, `release`,
  `supports`, and `version`.
- Repository fields are standardized around `canonical_repository`,
  `repository_url`, `repository_kind`, and `version_reference`.
- GitHub is no longer assumed to be the only valid repository host or repository
  kind.
